# 采用真转盘与命运抽签视觉语言

CONTEXT.md/README 一直把核心交互叫「转盘」，但实际下发的 `Wheel` 只是个 CSS loading spinner，旧注释还把「把抽中结果映射到扇区」推到 v2。本次把这个缺口补上：做一个真转盘——引擎把中签策略候选集的 ≈8 家样本（`wheelPool`，含中签家）随结果返回，扇区显示这些真实周边餐厅（截断名），减速停在中签家；并确立「命运抽签」视觉语言（暖米 `#FFF7ED` / 朱红 `#E2433B` / 金 `#E8A33D` / 深褐 `#2A1A14`，金色专用于中签与「就这家」锁定）为整条链路的设计语言。理由：产品卖的是「命运感」——消除选择困难、给一个就够了；spinner 给的是「网络加载中」的反面感受，真转盘 + 真候选才能把「在你周边真实选项里转命运」落到实处，金色锁定则给接受那一刻留了天然的情绪爆点。候选不足 ~4 家时跳过转盘直接揭卡（转一个选项不算命运，算通知）。

## 考虑过的替代

- **保留 spinner（转盘只是个词）**：零改动，但「转盘」名实不符，且 loading 感与「命运」相反。否决。
- **通用扇区（菜系/装饰图标，中签预定）**：不改引擎，但扇区不是用户真实选项，是善意的谎言。否决。
- **克制工具调色（维持冷灰 + 珊瑚）**：最安全，但命运感弱，转盘成了精致工具而非抽签。否决。
- **食欲温暖 / 扭蛋游戏调色**：前者稀释转盘的主角地位，后者默认暗色 + 小众口味、改动面过大。否决。
- **转盘形变成卡片（揭卡）**：最惊艳，但小程序里把 conic-gradient 扇区形变成卡片几何是深坑。否决，改用「落定金闪 → 卡片涌现」。

本 ADR supersede `src/components/Wheel/index.tsx` 中「mapping the pick to a wheel sector is a v2 UX refinement」的注释——真转盘不再是 v2 的事，是已决定、待实现。
