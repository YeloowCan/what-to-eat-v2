# PRD: 今天吃什么 — MVP

## Problem Statement

每到饭点，「今天吃哪家」成了高频折磨：周边餐厅太多，在美团/饿了么里翻不到头，比来比去耗时间，纠结半天要么随便凑合、要么干脆不吃了。选择困难的人需要的不是更多选项，而是**一个被直接给出的答案**——但现有的外卖平台都鼓励你「自己挑」，没人替你做这个决定。

## Solution

一个微信小程序：打开、授权定位，app 按你的位置 + 几个轻量**约束**（距离 / 营业中 / 菜系）从周边餐厅里**随机抽一家**给你。转盘停下的**候选展示**不满意就重抽，满意点「就这家」**落定为决策**。刚选过的店会被**冷却**短期排除，避免连续重复。无账号、无安装、无后端，用完即走。细节定不下来时，app 自动分层放宽（扩距离），实在没合适的才回头问你。

## User Stories

1. As 首次用户, I want 免安装直接在微信里打开小程序, so that 不用下载 App 就能用上
2. As 首次用户, I want 打开就明白这个 app 是干什么的, so that 不困惑
3. As 首次用户, I want 被引导授权定位, so that app 能找到我附近的餐厅
4. As 用户, I want 拒绝定位时 app 明确告知「需要定位才能用」并引导去开启, so that 我知道为什么用不了、怎么恢复
5. As 用户, I want app 用我的当前位置找周边餐厅, so that 结果是我真去得了的店
6. As 用户, I want 默认搜索半径合理(1km), so that 都是步行可达的店、不用我手动设
7. As 用户, I want 定位失败时清晰提示并可重试, so that 不卡在空白
8. As 用户, I want 能按距离筛选, so that 控制最远走多远
9. As 用户, I want 默认只看营业中的店, so that 永远不被推荐打烊的店
10. As 用户, I want 能按菜系筛选(火锅/川菜/快餐/日料/烧烤…), so that 跳过不想吃的品类
11. As 用户, I want 不设任何约束也能直接转盘, so that 完全交给随机
12. As 用户, I want 约束只在当次有效, so that 下次默认重置、不被绑死
13. As 选择困难的人, I want 一个转盘随机抽出一家餐厅, so that 直接给我一个答案、不用自己选
14. As 用户, I want 转盘有动画停在那家店, so that 有「命运感」、解压
15. As 用户, I want 看到候选展示的餐厅名/菜系/距离/营业状态, so that 判断这家行不行
16. As 用户, I want 不满意能一键重抽, so that 换一家试试
17. As 用户, I want 重抽很快, so that 不烦
18. As 用户, I want 重抽时被跳过的店不被记成决策, so that 历史只反映我真选过的
19. As 用户, I want 满意时点「就这家」确认, so that 定下来、去吃
20. As 用户, I want 确认后能跳去美团看详情/下单, so that 顺着就点了
21. As 用户, I want 我接受的决策被记住, so that 不会马上又推荐同一家
22. As 用户, I want 刚选过的几家短期不再出现, so that 不连续重复吃同一家
23. As 用户, I want 冷却自动生效, so that 不用我手动排除
24. As 用户, I want 附近符合条件的店太少时 app 自动扩大范围再抽, so that 不至于转不出东西
25. As 用户, I want 实在没合适的店时 app 提示我放宽菜系, so that 我自己决定要不要换品类
26. As 用户, I want 即使我指定了小众菜系也不被偷偷换成别的菜系, so that 我的明确意愿被尊重
27. As 回头用户, I want 看到我最近选过哪些餐厅, so that 回顾或复购
28. As 用户, I want 历史记录即使店后来关了也仍能显示当时选的是哪家, so that 历史不残缺
29. As 用户, I want 不注册不登录就能用, so that 零摩擦
30. As 用户, I want 不依赖联网后端也能用, so that 快、隐私好
31. As 用户, I want 高德数据拉取失败时友好提示重试, so that 不白屏
32. As 用户, I want 没有网络时给明确提示, so that 知道是网络问题而非 app 坏了
33. As 用户, I want 跳转美团失败时有兜底提示, so that 不卡死
34. As 用户, I want 从打开到能转盘很快, so that 饭点急的时候不磨蹭
35. As 用户, I want 转盘动画流畅, so that 体验好
36. As 用户, I want 被抽中的餐厅显示离我多远, so that 判断去不去
37. As 用户, I want 餐厅卡上能看到菜系标签, so that 快速确认品类

## Implementation Decisions

**载体与技术栈**
- 载体为微信小程序（ADR-0002）；用 Taro (React) 编写、编译为微信小程序（ADR-0003）。
- 无后端、匿名、无账号；所有状态只在 `wx.storage` 本地（设备绑定，不跨设备同步）。

**模块划分（逻辑模块，不含具体路径）**
- **决策引擎 (Decision Engine)** — 纯核心。公共接口：给定（位置、约束、冷却列表、POI 源端口、RNG 端口、时间端口）→ 产出**候选展示 (Suggestion)** 或「需用户放宽」信号；给定（被接受的 Suggestion）→ 产出**决策 (Decision)** 记录 + 更新后的冷却列表。引擎本身不碰任何 `wx.*` / 网络调用，全部依赖注入，故可纯函数测试。
- **位置端口 (Location port)** — 封装 `wx.getLocation`，返回坐标。MVP 拒绝授权即硬阻断，无手选兜底。
- **POI 源端口 (POI source port)** — 封装高德周边 POI 搜索；给定（坐标、半径、菜系 type）→ 餐厅列表（POI id、name、坐标、菜系、营业状态）。高德 key 用「小程序安全 Key + appid 白名单」方案，客户端直连。
- **决策存储端口 (Decision store port)** — 封装 `wx.storage`；读写决策历史（滚动保留最近 50 条）与冷却列表（最近 3 条已接受决策）。
- **深链端口 (Deep-link port)** — 封装 `wx.navigateToMiniProgram` 跳美团外卖小程序。MVP 不带 CPS 佣金参数，纯跳转。
- **UI 层** — Taro React 组件：定位权限硬阻断门、约束选择器、转盘视图、餐厅卡、接受/重抽控制、历史视图。

**领域与机制（遵循 CONTEXT.md 词汇与 ADR-0001）**
- 餐厅模型只持有身份（高德 POI id）、位置、菜系；**不含菜单/价格/下单**——这些是跳转美团后才发生的平台侧事实（ADR-0001）。
- 机制 = 筛选 + 均匀随机（MVP）；偏好加权留 v2。
- **候选集 = 周边 POI ∩ 约束 ∩ 冷却后剩余**。
- 约束 MVP 维度：距离（默认 1km）、营业中、菜系。其中**营业中与菜系为硬护栏**——候选集为空时分层放宽只动冷却与距离，这两项永不自动放宽。
- **冷却** = 最近 3 条已接受决策排除出候选集；重抽不进冷却。
- **承诺点** = 「就这家」。接受那一刻才写决策并快照；转盘停下只是候选展示，不算决策。
- 候选集为空时**分层放宽**：解冻冷却 → 扩距离 1→3→5→10km → 仍空则向用户发「需放宽菜系」信号；硬护栏永不自动放宽。

**Schema：决策记录 (Decision) 字段**
- 高德 POI id（去重与冷却依据）
- 餐厅名快照、坐标快照、菜系快照（落定时缓存，使历史不依赖后续反查高德——店关改名仍可显示）
- 接受时间戳
- 当次约束快照（距离值、菜系值）
- 不含菜单/价格

**API 契约**
- 高德周边 POI 搜索（客户端直连，安全 Key + appid 白名单）。
- `wx.getLocation`、`wx.navigateToMiniProgram`（美团外卖小程序）、`wx.storage`。

**关键交互流**
- 首次：打开 → 权限门 → 授权定位 →（可选设约束）→ 转盘 → 候选展示 → 重抽 or「就这家」→（可选跳美团）→ 决策落库 + 进冷却。
- 拒绝定位：全屏阻断，引导去系统设置开启授权；不可用。
- 候选集为空：引擎自动分层放宽并重抽；仍空则 UI 提示用户放宽菜系。

## Testing Decisions

**什么算好测试**：只测外部行为，不测实现细节。决策引擎通过其公共接口测——给定输入，断言可观测输出（抽出了哪个候选展示、是否触发了分层放宽、落了什么决策、冷却列表内容），不断言内部数据结构。

**单一自动化 seam：决策引擎**。注入 fake：POI 源（返回固定餐厅列表）、决策存储（内存版）、RNG（种子化 → 随机抽取可断言）、时间（固定）。这一个 seam 覆盖：
- 约束过滤（距离/营业中/菜系），含硬护栏行为
- 冷却排除最近 3 条已接受决策
- 从候选集均匀随机抽出候选展示（种子 RNG 可断言抽中哪家）
- 候选集为空时的分层放宽（先解冻冷却 → 扩距离 → 发「需放宽菜系」信号；营业中/菜系永不被自动放宽）
- 接受 → 生成带快照的决策记录 → 更新冷却 → 滚动保留 50 条
- 重抽不落决策、不进冷却

**不自动化的部分**：UI 层与所有 `wx.*` / 高德网络适配器——它们是薄胶水 + 环境耦合，自动化测试价值低、脆弱性高，MVP 用微信开发者工具手动验。

**先例**：无（greenfield）。此 seam 作为后续逻辑测试的约定先例。

## Out of Scope

- 群体决策（v2）
- 偏好加权抽取（v2）
- 用户主观标签（价格档 / 口味 / 场景）（v2，与加权一同引入）
- CPS 变现 / 美团联盟资质接入（v2，待拿到资质后给深链加佣金参数）
- 饿了么接入（v2，微信内跨生态跳转受限）
- 手选位置兜底（v2，定位被拒时可用）
- 多端（App / 支付宝小程序）（v2+，Taro 已为期权铺路，但非当前需求）
- 后端 / 账号 / 跨设备同步（v2）
- 菜单 / 价格 / 下单数据本地化（永不，见 ADR-0001）
