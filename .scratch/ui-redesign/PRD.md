# PRD: 今天吃什么 - 界面重做(真转盘 + 命运抽签视觉语言)

## Problem Statement

产品的领域语言一直把核心交互叫「转盘」--「转盘停下的候选展示」「转盘:过滤 + 均匀随机抽取」,卖的是「消除选择困难、给一个就够了」的**命运感**。但实际下发的 `Wheel` 只是个 CSS loading spinner(一个旋转圆环 + 「正在为你挑选…」),代码注释还把「把抽中结果映射到扇区」推到 v2。用户看到的不是命运在转,而是「App 在请求网络」--和产品想给的感受正好相反。与此同时,整个 UI 是一套没有章法的工具感:单一硬编码 `#ff5a5f` 散落四个文件、冷灰底、系统默认字体、唯一的「图标」是个 📍 emoji、没有设计系统、没有暗色、转盘->餐厅卡之间没有连续性、接受那一刻只有一句 toast。产品 promised 命运,交付的是加载中。

## Solution

按 ADR-0004,把「转盘」从隐喻做成实物,并确立一套「命运抽签」视觉语言作为整条链路的设计语言。具体:(1) 决策引擎把中签策略候选集的一个样本(`wheelPool`,必含中签家、上限 8)随**候选展示**返回,转盘扇区显示这些真实周边餐厅(截断名),减速停在中签家--命运在你周边真实选项里转,诚实不撒谎。(2) 转盘停下时中签扇区金闪 ~400ms,随后餐厅卡从中心 scale/上滑涌出、转盘淡出(落定->涌现)。(3) 点「就这家」时金色光晕外荡 + 「就这家✓」小戳 + 卡片收束 + 按钮形变成金色「去美团看看」(金色锁定)。(4) 重抽也重转,首抽 ~2.5s、重抽 ~1.2s。(5) 约束收成顶部一颗摘要 chip,转盘占主舞台,「最近选过」收成右上角小图标。(6) 调色 `#FFF7ED` 暖米底 / `#E2433B` 朱红主色 / `#E8A33D` 金(中签 & 接受)/ `#2A1A14` 深褐文,铺到卡片/约束/历史/导航栏/各状态,只做浅色。金色专用于「中签」与「就这家」两个关键时刻,守住「温暖笃定、不贴金」。

## User Stories

1. As 用户, I want 转盘真的在转、停在被抽中的那家店, so that 有真实的命运感、不是加载动画
2. As 用户, I want 转盘扇区显示我周边的真实餐厅, so that 知道命运是在我够得着的真实选项里转
3. As 用户, I want 转盘减速停下而非突然停, so that 有悬念、有「落定」的节拍
4. As 用户, I want 转盘停时被抽中扇区高亮闪金, so that 一眼看出落在了哪家
5. As 用户, I want 附近候选太少时不硬转转盘, so that 转一个选项不像命运而是通知
6. As 用户, I want 首次转盘有完整仪式感, so that 打开就有「交给命运」的体验
7. As 用户, I want 重抽也转盘但更快, so that 连抽不烦又有命运感
8. As 用户, I want 长店名在扇区上不被挤爆, so that 读得出
9. As 用户, I want 转盘停下后餐厅卡顺滑涌出, so that 从「命运落定」到「就是这家」是连续一气
10. As 用户, I want 餐厅卡清晰看到名/菜系/距离/营业状态, so that 判断这家行不行
11. As 用户, I want 餐厅卡上有菜系标签, so that 快速确认品类
12. As 用户, I want 看到被抽中餐厅离我多远, so that 判断去不去
13. As 用户, I want 看到营业状态, so that 知道是否还开着
14. As 用户, I want 引擎被迫放宽冷却或距离时有提示, so that 知道这家是「附近没更合适的、含刚选过的/已扩距离」
15. As 用户, I want 不满意一键换一家, so that 再试
16. As 用户, I want 满意点「就这家」确认, so that 定下来、去吃
17. As 用户, I want 点「就这家」时有金色的认定感, so that 「命运定了」有情绪收束
18. As 用户, I want 接受后能跳去美团看详情/下单, so that 顺着就点了
19. As 用户, I want 接受后被记住、短期不再推同一家, so that 不连续重复吃
20. As 用户, I want 约束默认不占主视觉, so that 转盘是主角
21. As 用户, I want 一眼看到当前约束摘要, so that 知道在什么条件下转
22. As 用户, I want 需要时一触展开完整约束, so that 改距离/菜系/营业中
23. As 用户, I want 改约束自动重转, so that 立刻看到新条件下的结果
24. As 用户, I want 约束当次有效, so that 下次默认重置、不被绑死
25. As 用户, I want 不设约束也能转, so that 完全交给随机
26. As 用户, I want 营业中与菜系是我说了算、永不偷偷放宽, so that 明确意愿被尊重
27. As 用户, I want 整个 app 视觉统一好看, so that 用着舒服、愿意天天用
28. As 用户, I want 暖色调有食欲和笃定感, so that 不是冷冰冰的工具
29. As 用户, I want 金色只用在「中签/就这家」关键时刻, so that 高光有分量、不滥用
30. As 用户, I want 导航栏和页面融为一体, so that 没有割裂感
31. As 用户, I want 字号/间距/圆角有一致节奏, so that 不杂乱
32. As 回头用户, I want 顺手看到最近选过哪些餐厅, so that 回顾或复购
33. As 用户, I want 历史入口不抢转盘主视觉, so that 转盘还是主角
34. As 用户, I want 历史即使店后来关了也仍显示当时选的, so that 历史不残缺
35. As 用户, I want 定位中有好看的等待状态, so that 不枯燥
36. As 用户, I want 定位失败时清晰提示并可重试, so that 不卡空白
37. As 用户, I want 网络出错时友好提示可重试, so that 知道是网络问题、能恢复
38. As 用户, I want 实在没合适的店时提示我放宽菜系, so that 我自己决定换品类
39. As 用户, I want 拒绝定位时明确告知并引导去开启, so that 知道为什么用不了、怎么恢复
40. As 用户, I want 跳转美团失败时有兜底提示, so that 不卡死
41. As 用户, I want 转盘动画流畅不卡, so that 体验好
42. As 用户, I want 从打开到能转盘很快, so that 饭点急的时候不磨蹭
43. As 用户, I want 在不同机型上都不错位, so that 都能用

## Implementation Decisions

**载体与技术栈(不变)**
- 微信小程序(ADR-0002)、Taro 4 (React) + TypeScript(ADR-0003)。无后端、匿名、无账号,状态只在 `wx.storage`。餐厅模型不含菜单/价格/下单(ADR-0001)。
- 本 spec 的所有决定遵循 **ADR-0004**(本次新增:真转盘 + 真候选 + 命运抽签视觉语言)。

**决策引擎(既有纯模块,扩展接口)**
- `pickSuggestion` 的公共结果中,suggestion 分支附带 `wheelPool`:中签策略候选集的一个样本,**必含中签家**、上限 `WHEEL_POOL_SIZE`。引擎仍为不碰 `wx.*`/网络的纯模块,全部依赖注入。
- 类型形态(编码决策比散文更精确):

  ```
  // PickResult 的 suggestion 分支新增 wheelPool:
  | { kind: 'suggestion'; suggestion: Suggestion; wheelPool: Restaurant[] }
  // wheelPool: 取自「产生中签家的那一个策略」的候选集样本,
  //           必含 suggestion.restaurant,上限 WHEEL_POOL_SIZE,顺序不保证(UI 按 poiId 定位中签扇区)
  ```

- 新常量 `WHEEL_POOL_SIZE`(初值 8)进 `constants.ts`,与既有 `COOLDOWN_WINDOW`/`HISTORY_LIMIT`/`DISTANCE_LADDER_KM` 同列。首抽与重抽的动画时长(~2.5s / ~1.2s)亦为可调初值。
- **「候选 <~4 家跳过转盘直接揭卡」是 UI 表现层决策,不进引擎**--引擎照实返回样本(候选少则样本就是全量),由 UI 按样本大小决定要不要跑转盘动画。
- 领域词汇与冷却/约束/分层放宽机制不变(CONTEXT.md 不动):候选集 = 周边 POI ∩ 约束 ∩ 冷却后剩余;营业中/菜系仍为硬护栏。

**spin 页编排(重写)**
- 从「fetch 与转盘动画赛跑(`Promise.all` 取 `MIN_SPIN_MS`)」改为:fetch -> 拿到候选(含 wheelPool) -> 转盘减速落定到中签扇区 -> 餐厅卡涌现 -> 接受/重抽。动画本身即「命运的节拍」,不再需要 min-duration 赛跑。
- 重抽复用同一转盘机制但提速(~1.2s vs 首抽 ~2.5s);衔接「卡片出 -> 转盘回 -> 快转 -> 落定 -> 卡片涌」。
- 候选 <~4 家跳过转盘直接揭卡;0 家走既有 `needsRelaxCuisine`。

**Wheel 组件(由占位 spinner 升级为真转盘)**
- conic-gradient 扇区 + 真实候选截断名(每扇一家、取前若干字) + 金色指针 + 减速缓动(cubic-bezier ease-out 带轻微回弹) + 落定金闪 ~400ms。揭卡后转盘淡出。supersede 旧注释「v2 UX refinement」(见 ADR-0004)。

**RestaurantCard(套调色 + 接受动效)**
- 命运调色:菜系标签朱红、营业状态绿/灰、放宽提示暖琥珀。
- 接受动效「金色锁定」:金色光晕外荡 + 卡头「就这家✓」小戳 + 卡片 scale 收束 + 底部按钮形变成金色「去美团看看」。

**ConstraintSelector(收成摘要 chip)**
- 顶部一颗可点摘要 chip(如「1km · 不限 · 营业中」),点开才展开完整选择器(距离 pills / 菜系 picker / 营业开关);改约束自动重转(沿用现有 onChange 行为)。

**历史入口与历史页**
- 「最近选过」从底部按钮收成右上角小图标;历史页套调色。历史快照机制不变(店关改名仍显示当时选的)。

**设计 tokens**
- `app.css` 加 CSS 变量(`--bg/--primary/--gold/--text` 等,以及圆角/间距/阴影),根治 `#ff5a5f` 等硬编码散落;全链路引用变量而非裸色值。

**导航栏与状态**
- `app.config.ts`:`navigationBarBackgroundColor` 改 `#FFF7ED`、`navigationBarTextStyle` 改 `black`,与暖米页面无缝。
- 各状态(定位中/定位失败/网络错/needsRelaxCuisine/空)套调色,暖而不刺眼;定位中用轻量金色动效替掉现有灰字。

**调色(命运抽签,浅色 only)**
- `#FFF7ED` 暖米底 / `#E2433B` 朱红主色 / `#E8A33D` 金(中签 & 接受)/ `#2A1A14` 深褐文。金色仅用于「中签」与「就这家」。不做暗色模式。

## Testing Decisions

**什么算好测试**:只测外部行为,不测实现细节。决策引擎通过公共接口测--给定输入,断言可观测输出(抽中了哪个候选展示、返回了什么 `wheelPool`、是否触发分层放宽),不断言内部数据结构。

**单一自动化 seam:决策引擎(既有,扩展)**。注入 fake:POI 源(固定餐厅列表)、决策存储(内存版)、RNG(种子化)、时间(固定)。在既有 seam 上**新增**覆盖:
- `wheelPool` 取自「产生中签家的那一个策略」的候选集(扩距离/解冻冷却后的宽池不算--诚实于实际抽签)
- `wheelPool` 必含中签家
- `wheelPool` 受 `WHEEL_POOL_SIZE` 上限
- 小候选集(不足上限)返回全量样本

**不自动化的部分**:全部 UI/视觉--转盘动画与减速落定、落定->涌现衔接、金色锁定接受动效、约束 chip 展开、命运调色与 tokens、导航栏、定位中/空/错误各状态。它们是表现层 + 环境耦合,用微信开发者工具手动验。沿用 MVP 的测试决策,不开新 seam。

**先例**:`engine/*.test.ts`(MVP 立下的引擎 seam 约定)。本 spec 仅在其上扩展,不引入第二个 seam--全仓库自动化 seam 仍只有一个。

## Out of Scope

- 暗色模式(本次只做浅色)
- 偏好加权抽取(v2)--届时转盘扇区权重才与偏好挂钩
- 群体决策(v2)
- 用户主观标签:价格档/口味/场景(v2,与加权一同引入)
- 后端 / 账号 / 跨设备同步(v2)
- 多端:App / 支付宝小程序(v2+,Taro 已为期权铺路)
- 菜单 / 价格 / 下单数据本地化(永不,ADR-0001)
- CPS 佣金 / 美团联盟资质(v2)
- 饿了么接入(v2)
- 手选位置兜底(v2,定位被拒时可用)
- Amap 鉴权改回 SDK + appid 白名单(独立改动,见 README「Amap 鉴权偏差」,不在本 spec)

## Further Notes

- 领域词汇见 `CONTEXT.md`(本次未改);架构决策见 `docs/adr/0001…0004`,其中 **ADR-0004 为本 spec 的依据**。
- MVP PRD 的 Further Notes 曾把「转盘 UX 细节(动画形态、首屏布局、餐厅卡精确字段)」标为「尚未敲定,留待实现时定」--本 spec 收口敲定此项。
- 调色为设计语言基底;具体色值/间距/圆角 tokens 在实现时落于 `app.css` 变量。`WHEEL_POOL_SIZE`、首抽/重抽动画时长为可调初值。
- 营业状态仍受高德数据限制(不可靠实时营业状态),为延续的已知限制,不在本 spec 解决。
- Amap 鉴权仍为 REST + 签名(弱于 PRD 原定的 SDK + appid 白名单),延续既有偏差,不在本 spec 范围。
